# AI Agent竞赛背景与提交规范摘要

本文档用于在两名参赛成员及各自 Codex 之间同步竞赛上下文。后续如题目理解、提交规范、评分细则、方案约束发生变化，应优先更新本文档，保证任何一方接手时能快速恢复上下文。

## 信息来源

- `GUIDANCE.md`：2026年ICT软件大赛参赛指导。
- `PLATFORM.md`：平台通用提交规范与评分方式。
- `01_llm_wiki/PLATFORM.md`：LLM-WIKI 赛题的专项提交规范。

## 竞赛背景

本次 ICT 软件大赛聚焦 AI 辅助研发，强调从真实业务场景出发，产出可运行、可评测、可复用的 Agent、Skill、插件、工具或其他自动化资产。

大赛分两个赛道：

- 赛道一：AI难题。可选择赛题组发布的业务难题，也可选手自拟题目。自拟题目可以围绕实际 AI 辅助研发难题设计，作品形式可以是 Agent、Skill、插件等。
- 赛道二：AI竞赛。由赛题组给出命题，参赛者围绕实际业务问题构建解决方案，重点考察 AI 技术运用、工程落地能力、实践沉淀和复用价值。

## 赛制要点

- 同一赛道下可以提交多个赛题作品，但该赛道最终只取最高分作品作为最终参赛作品。
- 每个赛题以最后一次成功提交并完成打分的结果为准。
- 平台允许多次提交，但以最后一次上传作品作为最终答题作品。
- 平台会提供客观评分，用于初筛、排序或评委参考。
- 决赛或答辩阶段会叠加主观评审，重点关注泛化能力、规范性、性能、应用价值等。

## 通用提交结构

作品以 zip 压缩包提交，命名格式：

```text
{赛道ID}_{题号}_{队名}.zip
```

示例：

```text
01_01_攻坚克难队.zip
```

解压后应以同名文件夹作为根目录。关键目录和文件如下：

```text
/INSTRUCTION.md
/work
/result
/result/output.md
/logs
/logs/interaction.md
/logs/trace
```

如提交 Skill，应符合：

```text
/work/skills/{your-skill-name}/SKILL.md
```

如提交 subagent，可放在：

```text
/work/skills/{your-subagent-name}.md
```

如赛道一选择自拟题目，还必须包含：

```text
/problem_statement/PROBLEM.md
```

且题号固定填写 `00`。

## INSTRUCTION.md 要点

`INSTRUCTION.md` 是平台自动评测的入口说明书。平台会读取它来部署、启动、执行作品，并获取结果。

必须写清楚：

- 环境准备：依赖安装、编译构建、服务启动、Docker/MCP/Skill/Agent 安装等。
- 执行方式：启动命令、执行命令、参数说明、调用方式、执行顺序。
- 执行完成判定：命令退出、服务状态、指定文件生成、接口返回成功等。
- 结果获取方式：修复后的工程位置、输出文件位置、日志文件位置、报告文件位置等。

重要约束：

- 所有步骤必须能够自动完成。
- 不得依赖人工交互。
- 必须能明确判断执行完成。
- 必须能明确获取最终结果。
- 不应修改 REST API 契约。
- 不应硬编码验证环境或针对公开样例投机。
- 应提供方案说明与验证方案。

## 评分规则

普通赛题通常包含客观评分与主观评分。

客观评分关注准确性与稳定性：

- 准确性：5 个用例集里通过用例数最高的一次。
- 稳定性：5 个用例集都通过的用例交集。
- 平台专项文档给出的最终客观得分公式：

```text
最终得分 = 100 * (稳定通过的用例数 + 准确通过的用例数) / (用例总数 * 2)
```

参赛指导中也提到客观评分可拆为：

- 准确性 30 分。
- 稳定性 30 分。

主观评分通常关注：

- 泛用性：是否能推广到同类场景，是否存在针对验证环境硬编码。
- 规范性：是否符合 Skill、SPEC、MCP 或相关交付规范。
- 性能：运行时长、Token 消耗、均值、最大值、最小值等。

赛道一自拟题目的客观评分主要用于辅助评委判断准确性和稳定性，不参与客观评分中位值排序，综合得分更依赖题目应用价值和难度的主观评定。

## 当前选题理解：01_01_llm_wiki

本队当前选择 `01_llm_wiki` 赛题，提交根目录约定为 `01_01_teamname/`，最终压缩该目录提交。

题目目标是构建一个自动化的 LLM-WIKI 问答/操作系统。平台会在 `work` 同级释放 `llm-wiki/`，其中包含 `docs/`、`question/group-*.md`、`output/`、`Permission.json` 等。作品需要读取每组 JSON 数组格式的问题，检索和处理知识库文件，并按标准格式生成 `llm-wiki/output/group-*-answer.md`。

核心能力包括：

- 多格式文件解析：处理 200+ 文件，覆盖 `doc`、`docx`、`ppt`、`pptx`、`xls`、`xlsx`、`xml`、`java`、`py`、`html`、`md`、`js` 及其他文本类文件。
- 批注/TODO 管理：识别办公文件批注、代码 TODO，支持按责任人、日期等统计、筛选，并按批注要求修复文件，修复结果输出到 `llm-wiki/output/fixed/`。
- 知识库问答：回答文件数量、文件路径、业务资料、环境命令、代码执行结果、Excel 透视图等问题。
- 安全防护：对系统密码/数据库密钥/越权文件读取/高危命令/Prompt 注入/命中 `Permission.json` 黑名单的请求，统一返回 `{"error_msg":"高危命令，拒绝访问"}`。但 `02_环境信息` 下允许检索的环境密码不应被误杀。

答案格式必须严格匹配题目规范：答案是 JSON 数组；每项包含原问题 `id` 和 `answer`；常见 answer 包括 `{"datas":[...]}`、`{"count":n}`、`{"doc":n}`、`{"source":"...","target":"..."}`、`{"error_msg":"高危命令，拒绝访问"}`。路径统一从 `docs/` 或 `output/` 开始。

公开样例只包含少量题目，隐藏评测会替换问题和文档，因此后续实现必须泛化解析和判定逻辑，不能针对样例硬编码。

## 当前工作原则

- 先把赛题理解、输入输出、评测方式和隐藏用例风险梳理清楚，再定方案。
- 方案要优先追求稳定、泛化、可自动执行，而不是只跑通公开用例。
- `INSTRUCTION.md` 要写成裁判可直接执行的自动化说明。
- 交付结构必须完整：`work/`、`result/`、`logs/`、必要时 `problem_statement/`。
- 所有重要决策、题目理解、方案取舍和进度都应同步到 `codeX_assist` 下的共享文档。
