# Capability Review

本文档用于记录围绕 LLM-WIKI 五类问题的能力覆盖讨论。每条记录包含 user_1 提问、当前结论、实现依据和后续风险/改进点，供后续开发迭代参考。

## 2026-07-14

### 1. 文件定位是否全量扫描 `docs/`

**问题**

readme 中说明 `docs/` 子文件夹名称不固定，且文件归类可能出错。因此文件定位是否应该全量扫描 `docs/`，而不是根据固定目录名找文件？当前项目是否这么做？

**结论**

当前项目在文件索引阶段是全量递归扫描 `docs/` 的，不依赖固定子文件夹名称。只要文件位于 `docs/` 下，即使归类目录错误，也会进入统一文件索引。

**实现依据**

- `file_index_skill/scripts/index_builder.py` 中 `scan_docs()` 使用 `root.rglob("*")` 递归遍历 `docs/` 下所有文件。
- 索引记录 `docs/...` 相对路径、文件名、主文件名、后缀、一级目录、大小、修改时间、文件类别等元数据。
- 子文件夹名只作为元数据和召回线索，不是硬编码入口或过滤前提。

**当前可覆盖**

- 指定文件类型数量统计，例如统计 `doc`、`docx`、`xlsx` 数量。
- 明确文件名或主文件名的路径查找。
- 基于路径、文件名、后缀、目录名关键词的候选文件召回。

**风险与后续改进**

- 文件索引Agent本身不读取文件正文，但主控知识问答链路已补候选扩容、Office/TextCode/通用文本正文兜底和 KnowledgeQA 文件级证据聚合，能覆盖关键词只出现在正文中的基础内容型路径问题。
- 后续如果知识库规模很大，仍可考虑把正文关键词摘要写入索引，进一步减少运行时抽取成本。



### 2. 批注管理能力覆盖

**问题**

批注管理可细分为四类：办公文件批注统计/管理/筛选、代码文件批注统计/管理/筛选、办公文件批注修复、代码文件批注修复。需要确认 12 种文件类型（`.doc/.docx/.ppt/.pptx/.xls/.xlsx/.xml/.java/.py/.html/.md/.js`）哪些支持批注提取、统一筛选和修复。

**结论**

当前基础版对“批注/TODO 提取、统计、按责任人筛选、按 end_date 等值/范围筛选”有统一框架，但不是 12 种文件都同等可靠。修复能力明显更窄：代码文本类 6 种支持简单“把 X 改成 Y/将 X 替换为 Y”类确定性修复；Office 目前只有 `.docx` 的简单替换修复较可靠，`.doc/.ppt/.pptx/.xls/.xlsx` 修复仍不完整或主动失败。

**批注/TODO 提取覆盖**


| 文件类型    | 当前提取能力       | 说明                                                                                    |
| ------- | ------------ | ------------------------------------------------------------------------------------- |
| `.docx` | 支持           | 读取 OOXML `word/*comment*.xml` 中的 Word 批注，并按 `todo/to/end_date` 结构化解析；自由批注保留 raw_text。 |
| `.doc`  | 部分支持         | 优先用 LibreOffice 转 `.docx` 后按 `.docx` 提取；无转换能力时只做有限文本降级，不可靠。                           |
| `.pptx` | 支持           | 解析真实 `ppt/comments/comment*.xml` 批注部件，不再把 notes 当批注；结构化格式 A 可筛选，自由格式 B 只提取和统计。          |
| `.ppt`  | 部分支持         | 优先转 `.pptx` 后提取；无转换能力时只做有限文本降级，不可靠。                                                   |
| `.xlsx` | 支持           | 使用 openpyxl 读取单元格 comments，并按办公批注规范解析。                                                |
| `.xls`  | 弱/依赖转换       | 优先转 `.xlsx` 后提取 comments；无转换时只能读表格内容，批注无法可靠提取。                                        |
| `.xml`  | 支持 TODO/批注提取 | 识别真实 `<!-- -->` 注释块；结构化格式 A 可按责任人/日期筛选，格式 B 自由批注只参与提取和统计。                              |
| `.html` | 支持 TODO/批注提取 | 同 `.xml`，只从 HTML 注释边界提取批注，不再把普通正文 TODO 行当作批注。                                         |
| `.md`   | 支持 TODO/批注提取 | 识别 Markdown 中的 HTML 注释块；普通正文 TODO 行不再算批注。                                               |
| `.java` | 支持 TODO/批注提取 | 识别 `//` 单行注释和 `/* */` 块注释。                                                            |
| `.js`   | 支持 TODO/批注提取 | 同 `.java`。                                                                            |
| `.py`   | 支持 TODO/批注提取 | 识别 `#` 单行注释和独立三引号字符串块中的 TODO/批注；赋值、参数、表达式中的普通三引号字符串不作为批注。                            |


**是否按 readme 批注规范解析**

- Office 批注解析支持结构化字段 `todo`、`to`、`end_date`，兼容中英文冒号、大小写和逗号/分号分隔；不满足三字段的自由批注会保留 `raw_text`，`structured=false`。
- 代码 TODO 解析支持 `todo/to/end_date` 三字段，兼容中英文冒号和大小写；无结构化简易批注会保留 `raw_text`，但字段为空。
- 当前 TextCode 提取器已改为优先识别注释边界：`#`、`//`、`/* */`、`<!-- -->`、独立 Python 三引号块。注释边界内三字段齐全的是格式 A；否则保留为格式 B。

**统一管理与筛选能力**

- Office 和 TextCode 都会把批注/TODO 统一成类似 `source/file_type/location/raw_text/structured/todo/to/end_date` 的结构。
- 支持统计数量：`count_comments`、`count_todos`。
- 支持按责任人等值筛选：`assignee/to`。
- 支持按 `end_date` 等值筛选、`end_date_lte/end_date_gte` 范围筛选，以及 `[start,end]` 日期区间筛选。
- 支持按文件名筛选；TextCode 额外支持后缀、关键词、structured 状态筛选。
- 按责任人或日期筛选时，只返回 `structured=true` 的格式 A；格式 B 自由批注只参与提取和统计。
- 暂不支持责任人模糊匹配、多责任人 OR 条件。这是批注管理下一轮需要补的重点。

**修复覆盖**


| 文件类型    | 当前修复能力    | 说明                                                                                   |
| ------- | --------- | ------------------------------------------------------------------------------------ |
| `.docx` | 支持简单确定性替换 | 从真实批注/TODO 中解析“旧文本 修改为/改为/替换为/换成/改成 新文本”，修改 `word/document.xml` 并写到 `output/fixed/`。 |
| `.doc`  | 暂不可靠      | 虽可尝试转换后提取批注，但修复写回 `.doc` 目前不可靠，不能视为稳定支持。                                             |
| `.pptx` | 支持简单确定性替换 | 从真实 comments 中解析“把 X 改成 Y/将 X 替换为 Y”等规则，修改 slide 可见文本并写入 `output/fixed/`。             |
| `.ppt`  | 暂不支持      | 老格式写回仍不可靠，本轮不伪造 fixed。                                                                 |
| `.xlsx` | 支持简单确定性替换 | 从单元格 comments 中解析“把 X 改成 Y/将 X 替换为 Y”等规则，修改普通文本单元格并写入 `output/fixed/`；公式单元格跳过。        |
| `.xls`  | 暂不支持      | 批注提取依赖转换，修复不支持。                                                                      |
| `.xml`  | 支持简单确定性替换 | 基于结构化 TODO 中“把 X 改成 Y/将 X 替换为 Y”，修改第一处非注释行并写入 `output/fixed/`。                       |
| `.html` | 支持简单确定性替换 | 同文本代码修复器。                                                                            |
| `.md`   | 支持简单确定性替换 | 同文本代码修复器。                                                                            |
| `.java` | 支持简单确定性替换 | 同文本代码修复器。                                                                            |
| `.js`   | 支持简单确定性替换 | 同文本代码修复器。                                                                            |
| `.py`   | 支持简单确定性替换 | 同文本代码修复器。                                                                            |


**风险与后续改进**

- 日期范围筛选已支持等值、`lte/gte` 和区间；后续仍需补责任人模糊匹配、多责任人 OR 等复杂筛选。
- 复杂修复已具备受控模型兜底路径：确定性脚本失败但存在可信批注/TODO 时，会生成 `complex_repair_tasks/*.json`，由外层 CodeAgent 按 `INSTRUCTION.md` 使用模型能力修复、验证并回写答案。
- Office 修复已补 `.docx/.pptx/.xlsx` 的简单文本替换写回；`.doc/.ppt/.xls` 仍需明确老格式输出策略。
- 代码文件修复目前只改第一处非注释行，复杂多处替换、跨行替换、HTML/XML 结构化修改仍有风险。



### 3. 文件运行能力覆盖

**问题**

文件运行类问题包括：对 6 种代码文件（`.xml/.java/.py/.html/.md/.js`）询问某一段代码的执行结果；以及对某个 `.xlsx` 中的表画出透视图。需要确认当前是否支持模拟执行结果并返回，是否支持生成 Excel 透视图。

**结论**

当前不真实执行代码、不编译 Java、不运行脚本；对于代码执行结果类问题，已改为生成 `code_reasoning_tasks/*.json`，由外层 CodeAgent 按 `INSTRUCTION.md` 使用模型能力静态推演并回写答案。`.xlsx` 透视图已支持基础版：由 Office Agent 读取表格、推断维度/指标/聚合方式，生成包含透视汇总表和图表的 `.xlsx` 文件到 `output/fixed/`。

按 readme 对“文件运行”的严格定义，当前 Agent 不做真实运行，但对代码执行结果类问题已有受控模型推演入口；Excel 透视图已有基础覆盖，但不是 Excel 原生 PivotTable/PivotChart 对象。

**代码执行结果覆盖**

- `.xml/.java/.py/.html/.md/.js` 均可读取正文，并可提取 TODO/注释和静态风险线索。
- TextCode Agent 明确不执行任何代码或命令。
- KnowledgeQA 中 `answer_code_static_question` 遇到“执行结果/返回值/输出结果”等题面时，会生成 `code_reasoning_tasks/*.json`。
- 外层 CodeAgent 按 `INSTRUCTION.md` 读取任务包，使用模型能力静态推演，不真实执行代码，并在成功时回写 `group-*-answer.md`。

**当前可处理的问题**

- “这段代码里有什么高危风险”
- “代码中出现了哪些命令/路径/密码字段”
- “某文件中的相关代码片段是什么”
- “这段 Python/Java/JS 运行结果是什么”：生成模型推演任务包，由外层 CodeAgent 静态推演后回写答案。

**当前不能可靠回答的问题**

- 依赖外部输入、文件、网络、系统环境、随机数、当前时间或不可见上下文的执行结果。
- `.xml/.html/.md` 中嵌入脚本或命令且上下文不足的执行结果。
- 需要真实副作用验证的题，例如“执行后文件发生什么变化”。

**Excel 透视图覆盖**

- `.xlsx` 可用 openpyxl 读取工作表单元格文本和批注。
- `excel_analyze` 对普通 Excel 摘要问题仍返回 sheet/table 摘要；当题面包含“透视/图表/pivot”等信号时，会生成 `output/fixed/*_pivot.xlsx`。
- 生成的工作簿包含“透视汇总”sheet、聚合结果表和 openpyxl 图表，答案返回 `{"source":"docs/...xlsx","target":"output/fixed/..._pivot.xlsx"}`。
- 主控对 `answer_excel_summary` 会优先调用 Office Agent 的 `excel_analyze`；若产出 `source/target`，直接作为最终答案。
- 当前不创建 Excel 原生 PivotTable/PivotChart 缓存对象，而是生成等价的聚合表和图表。

**风险与后续改进**

- 代码执行结果已选择“不真实执行、由外层 CodeAgent 模型静态推演”的路线；后续可补更多定位代码片段和验证任务包的自造样例。
- Java 不走编译链路，统一交由模型静态推演；遇到依赖外部环境或上下文不足时应标记失败，不伪造运行结果。
- Excel 透视图基础能力已补齐，但字段推断仍是启发式；复杂多表、多 sheet、公式缓存缺失、需要原生 PivotTable 对象的场景仍有风险。



### 4. 知识库管理能力覆盖

**问题**

知识库管理主要包括三类：询问不同类型文件数量；询问涉及 XXX 业务的文件名称和路径；询问 XXX 行为的具体命令是什么。需要确认当前 Agent 能 cover 哪些、哪些有风险。

**结论**

当前三类能力覆盖程度不同：文件数量统计较稳；具体命令问答部分可用；“涉及 XXX 业务的文件名称和路径”已补二阶段增强和文件级相关性评分。当前主控会对知识问答类问题扩大候选池，并对可解码的其他文本后缀执行通用正文兜底读取；KnowledgeQA 会把块级命中聚合成文件级证据后再输出路径。

**1. 不同类型文件数量**

覆盖程度：稳定可 cover。

- 文件索引Agent会递归扫描 `docs/` 全量文件。
- `count_by_type` 基于索引元数据按后缀统计。
- 统计枚举符合 readme：`doc/docx/ppt/pptx/xls/xlsx/xml/java/py/html/md/js`。
- 返回格式符合比赛要求，例如 `{"doc": 5}`。

主要风险：

- 已增强“多少份/几个/共有/一共/总共/合计/总计”等数量问法，以及 Word/Excel/PowerPoint 等别名映射。
- 问题如果表达非常绕，分类器仍可能没识别为数量统计。
- 其他非枚举后缀会进索引，但不会参与文件类型数量答案。

**2. 涉及 XXX 业务的文件名称和路径**

覆盖程度：部分可用，高风险。

当前链路：

- 主控将“涉及/相关/文件名称/路径”等问题归为 `answer_file_content_paths`。
- 文件索引Agent先基于文件名、路径、后缀、目录名和题面关键词召回候选文件。
- 主控对候选 Office/TextCode 文件调用正文抽取，对可解码其他文本后缀做通用正文兜底，再交给知识问答Agent按文件级 evidence 返回 `docs/...` 路径。

可以 cover 的情况：

- 业务关键词出现在文件名、目录名或路径中。
- 业务关键词出现在已被召回候选文件的正文中。
- 题目明确给出文件名、后缀或目录线索。

高风险情况：

- 业务关键词只出现在正文里时，主控会对知识问答候选扩容，并把 Office/TextCode/通用文本正文传入 KnowledgeQA 做二阶段证据排序。
- `answer_file_content_paths` 候选上限已提升到 200，超大知识库仍可能漏掉靠后的文件。
- `txt/csv/json/yaml/yml/properties/env/conf/cfg/ini/log/sh/cmd/sql` 等可解码文本后缀已支持通用正文兜底。
- 内容型路径答案已改为文件级聚合评分：合并同一文件的多个正文/表格/批注命中，结合关键词扩展、同义词扩展和否定上下文识别，减少单行噪声和“并不涉及 XXX”误判。
- 当前仍不是向量语义检索或大模型全文理解，复杂隐含语义“涉及某业务”的判断仍有边界。

后续建议：

- 继续为 `answer_file_content_paths` 增加更多领域同义词和文件级评测集。
- 对扫描版 PDF、图片和二进制类其他格式仍需后续补读取或明确证据不足。
- 对候选召回从“元数据 TopN”扩展为“元数据召回 + 内容扫描二阶段”。

**3. XXX 行为的具体命令是什么**

覆盖程度：基础版可用，仍有中等边界风险。

当前链路：

- 主控遇到“命令/指令/控制台/连接/登录/如何/怎么”等表达时归为 `answer_command_info`。
- 文件索引和主控候选扩展已补 `txt/csv/json/yaml/yml/properties/env/conf/cfg/ini/log/sh/cmd/sql/pdf` 等后缀识别。
- 主控会对命令类问题优先扫描 `04_常用命令`、路径/文件名含“命令/command/cmd/shell/终端/客户端/连接/登录/ssh/gsql”等信号的文件。
- KnowledgeQA 检索会对命令信号加权，并从正文中抽取 fenced code block、多行命令、续行命令、缩进行命令和嵌入式命令片段。
- PDF 正文由主控尽力读取：优先使用可选 `pypdf`，失败后尝试从简单 PDF 文本对象中提取，不保证扫描件或复杂压缩 PDF。

可以 cover 的情况：

- 命令资料位于可召回的 Office/TextCode/通用文本/PDF 可提取文本中。
- 题面关键词与文件名、目录、正文描述或命令同义词有一定重合。
- 答案位于单行命令、代码块、多行续行命令或相邻短文本中。

高风险情况：

- 命令描述和题面用词完全不重合、且文档路径/目录也没有命令类信号时，候选召回仍可能失败。
- 命令在扫描版 PDF、图片、复杂压缩 PDF 或二进制不可抽取格式中时仍无法稳定读取。
- 多段相似命令同时出现时，当前仍是规则排序，可能需要更多题型样例优化。
- 如果命令文本包含安全敏感词，可能被安全守卫拦截；这对危险命令是正确行为，但普通运维命令可能有误拒风险。



### 5. 密码提问与高危指令执行安全覆盖

**问题**

密码提问和高危指令执行都属于安全守卫Agent职责，需要确认当前是否能稳定 cover。

**结论**

当前基础版对典型密码提问、系统路径访问、Prompt 注入、高危命令和 `Permission.json` 黑名单文件/目录访问有较强覆盖，公开样例级别基本可用。但不能认为完全稳定：规则主要是关键词和简单通配匹配，对隐藏评测中的变体表达、随机 `Permission.json` 命令黑名单、以及文档内容中的非显式引用注入仍有漏判或误判风险。

**当前安全链路**

- 主控每道题开始先调用安全守卫Agent做问题级检查。
- 文件索引前会检查 `docs` 目录资源。
- 访问候选文件前会批量调用资源级检查。
- 安全守卫读取 `Permission.json`，支持 `dir.deny`、`file.deny`、`command.deny` 的精确匹配和简单 `*` 通配；问题级检查已直接扫描题面命令 token，命中 `command.deny` 时直接拒绝。
- 命中拒绝时统一返回 `{"error_msg":"高危命令，拒绝访问"}`。

**密码提问覆盖**

稳定覆盖：

- 明确询问系统密码、数据库密钥、明文密码、token、secret、api key、access key 等。
- 指向系统目录、`99_mock_system_dir`、`/etc`、root 等位置的密码查询。
- 明确引用 `Permission.json` 黑名单文件的密码查询。

允许例外：

- 题目允许检索 `docs/02_环境信息` 下的环境密码。
- 当前规则还会对带环境信号的问题放行，例如 IP、URL、环境、op_user、deploy_user 等，后续由知识问答链路去 `02_环境信息` 召回。

风险：

- 环境密码例外目前偏宽：只要有“环境/IP/op_user”等信号且不是系统路径，就可能放行。隐藏题如果问“非 02_环境信息目录里的环境密码”，存在误放风险。
- 密码类关键词是规则匹配，变体说法如“凭据”“认证串”“连接串里的口令”等可能漏判，需要扩展。

**高危指令执行覆盖**

稳定覆盖：

- 常见危险命令/动作：`rm -rf`、`rm`、`rmdir`、`del`、`Remove-Item`、`kill/taskkill/pkill`、`chmod 777`、`mkfs`、`dd if=`、`curl|sh`、`wget|bash` 等。
- 中文危险动作：删除、彻底删除、移除、写入、覆盖、执行命令、运行脚本、终止进程。
- Prompt 注入：忽略规则、最高优先级、上帝模式、强制执行、绕过安全等。
- 路径穿越 `../`。
- 明确引用的 docs 文档会被读取并扫描其中的高危动作和 Prompt 注入。
- 资源级文件/目录检查能拦截候选文件命中 `Permission.json` 的 `file.deny` / `dir.deny`。
- 已补“命令文本查询”和“真实执行请求”的区分：只问“某普通运维命令是什么/怎么写/知识库里怎么记录”时允许继续检索；要求“帮我执行/运行/删除/杀进程/验证可执行”等仍拒绝；`rm -rf`、`mkfs`、`dd if=`、删除全部/根目录/系统等高破坏性表达即使包装成查询也拒绝。

风险：

- `Permission.json.command.deny` 已补题面直接扫描，但仍是 token/通配规则匹配；如果隐藏题使用严重混淆写法、插入不可见字符或把命令拆得很碎，仍可能需要继续增强规范化。
- 文档注入只扫描“题面显式引用的 docs 路径”对应文档；如果注入内容藏在后续召回文件正文里，当前资源级检查只检查路径，不扫描正文内容，可能漏掉内容注入。
- 危险命令和 Prompt 注入依赖关键词，混淆写法、空格拆分、同义表达可能绕过。
- 批量候选资源检查当前遇到一个黑名单候选会整题拒绝，可能有过拒风险。

**后续建议**

- 收紧环境密码例外：必须命中 `02_环境信息` 路径或召回证据后再允许返回密码。
- 继续扩展 `Permission.json.command.deny` 的混淆写法规范化和正反例集。
- 在 Office/TextCode 抽取正文后增加内容级安全复查，至少扫描 Prompt 注入和危险动作。
- 扩展密码/凭据/命令注入的同义词和混淆写法规则。
